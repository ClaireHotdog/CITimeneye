Timeneye Restructuring — Reference Guide & Project Proposal

Corporate Insight · 2026 Time Tracking Redesign · Phase Taxonomy, Naming Standards & Metadata Strategy

Updated March 2026
This document proposes a full restructuring of Corporate Insight's Timeneye time tracking system. The current setup has grown organically over several years into a structure that makes accurate billing, reporting, and team planning unnecessarily difficult. The proposal below outlines the specific problems found, the new system design, and the measurable benefits it delivers.
The Problem — By the Numbers
313

Many are duplicates, legacy entries, or projects that should have been archived years ago. Staff must scroll through hundreds of options to find the right project.

Nearly 1,000 different phase names across the system — many doing the same thing. Some projects had 50+ phases. This makes cross-project filtering completely unreliable.

~$0

Because billable vs. non-billable classifications were inconsistent and phases didn't map to billing categories, true billing reports couldn't be trusted.
Root Causes — What Went Wrong
⚠️ Before — The Old Structure
❌
Phases were used as a catch-all — "Project Work," "Admin," "Other" explained nothing and provided zero reporting value
❌
Date-specific phase names like "January Report" or "12-12-2025 Screenshots" made phases undiscoverable and non-reusable
❌
No consistent distinction between billable and non-billable work at the phase level — billing audits required manual entry-by-entry review
❌
Project names mixed abbreviations ("BNK Monitor"), date formats ("'26"), and separators (pipes without spaces), making alphabetical sorting unreliable
❌
Tags were undefined — staff invented their own tags, creating 36 different tag values with no standardization or documented purpose
❌
No phase categories to group billable vs. non-billable work — every billing check required opening individual entries
❌
Internal overhead time was scattered across dozens of projects and phases, making it impossible to calculate true overhead rates
❌
No color-coded project types — staff had to read every project name to determine if it was a monitor, internal task, or client request
✅ After — The New Structure
✅
Phases represent workflow stages only — standardized names like "Content Production," "Quality Review," "Final Delivery" work across all similar projects
✅
Dates belong in task/todo names, never phase names — phases are evergreen and reusable year after year
✅
Every phase belongs to a phase category that is explicitly Billable or Non-Billable — billing audits take seconds, not hours
✅
All project names follow a single formula per project type — full name, title case, 4-digit year, consistent pipe formatting
✅
14 approved tags with documented purposes, organized by Contributing Team, Channel, and Flag categories — no freelancing
✅
Phase categories enable one-click billing audits: filter Non-Billable category + Billing Status = Billable → instantly find every misclassification
✅
All internal work consolidates into gold/yellow projects by team — overhead is measurable and reportable in real time
✅
7 project colors map to project types — staff can visually navigate to their work without reading every project name

Implementation Approach

  1. Phase 1 — Archive: Archive all projects that have had no entries in 12+ months. Keeps the active list clean without losing historical data.
  2. Phase 2 — Rename: Rename all active projects to follow the new naming formulas. Use bulk edit in Timeneye admin where possible.
  3. Phase 3 — Phase Consolidation: Map existing phases to the ~22 standardized phases per project type (see Phase Mapping tab). Migrate entries in bulk.
  4. Phase 4 — Tag Cleanup: Standardize tags to the 14 approved values. Remove or merge non-standard tags, then re-tag entries where coverage is missing.
  5. Phase 5 — Rollout & Training: Distribute this reference guide to all staff. Hold a brief walkthrough with each team. Update the Timeneye new-entry templates.
Ongoing Maintenance
A well-structured system degrades without active stewardship. The restructuring work described above creates a clean foundation — but only ongoing maintenance keeps it that way. The tasks below represent the recurring administrative work required to keep the system accurate, navigable, and trustworthy over time.
Admin Responsibilities
📊 Reporting
Run and distribute weekly time summary reports to team managers — hours logged by project, phase, and person
Produce monthly billing reports showing total billable hours per project and client, segmented by phase category
Generate overhead and utilization snapshots — ratio of billable to non-billable time by team and individual
Deliver year-over-year comparisons at each annual rollover, tracking productivity trends across the same projects
✅ Billing QA & Audits
Run regular billability audits: filter non-billable phase categories for entries marked Billable (and vice versa) — correct mismatches in bulk
Flag entries logged under incorrect projects — e.g. billable client work accidentally logged under internal projects
Identify entries with no phase or entries using deprecated phase names that slipped through
Audit tag coverage — check that contributing team and work channel tags are being applied where expected, and investigate gaps
⏰ Time Logging Compliance
Monitor the entries log for gaps — identify team members who haven't logged time in the current week or billing period
Reach out to staff who are behind on logging, with a reminder of the deadline and the correct project/phase to use
Escalate persistent non-compliance to the relevant manager if reminders aren't actioned
Track compliance trends over time to identify teams or individuals who consistently lag, and adjust outreach accordingly
🗂️ Project & System Upkeep
Create new projects when contracts are signed — name, color-code, assign phases, and set the correct client and billing defaults
Add users to projects as staff join new teams, start working on new clients, or change roles
Archive completed projects at year-end or when a contract ends — keeps the active list navigable without deleting historical data
Rename or reconfigure projects when client names, product names, or contract scopes change
Catch and correct naming convention violations before they proliferate — new projects created by non-admins may not follow standards
📚 Documentation & Training
Maintain and update this reference guide as the system evolves — add new FAQs, update project lists, revise naming examples
Serve as the first point of contact for staff who don't know where to log time, need a new project, or encounter system issues
Onboard new hires to the Timeneye system — walk through logging expectations, project structure, and how to read their entries
Update documentation whenever system changes occur — phase additions, tag updates, new project types, or policy shifts
🔁 Annual Rollover
Create all new-year project copies in December/January — duplicate active projects with updated year, reset phases and membership
Archive all prior-year projects to remove them from the active list without deleting historical entries
Verify that WBS codes and Salesforce opportunity references in project descriptions are updated to reflect new contract years
Confirm that phase lists and tag lists still match the standard taxonomy — roll over any approved changes from the prior year's review

Why Ongoing Maintenance Matters

The original Timeneye system degraded to 964 unique phase names, 42% missing tags, and unreliable billing data gradually, over time — the result of no active upkeep. The same entropy will return without continued attention: staff create ad-hoc phases, naming conventions drift, billability classifications go unchecked, and the system becomes too noisy to trust.

The maintenance tasks above are not optional extras — they are what separates a one-time cleanup from a durable, reliable system. The restructuring builds the foundation. Regular upkeep keeps it standing.

The new Timeneye structure uses ~22 standardized phases across six project types. Phases now represent workflow stages — not deliverable names, dates, or client names. This makes every phase searchable and filterable across the entire system.

Phase Categories are the backbone of the billing audit system. Every phase belongs to exactly one category, and every category is permanently set to either Billable or Non-Billable. This means a single filter on Phase Category instantly reveals whether time was logged correctly — no manual entry-by-entry review needed.
Billable Categories
Billable — Client Deliverables
Applies to: Monitor Products · Custom Projects · Benchmarks
Core production phases: Planning, Research, Content Production, QA, and Delivery. These phases represent the primary work product clients pay for.
Filter use: Phase Category = "Billable — Client Deliverables" + Date Range → Total billable production hours for client invoicing
Billable — Research & Data
Applies to: UX Research Projects
All UX research phases from Study Planning through Client Presentation. These are fully billable even though they're a distinct methodology from monitor work.
Filter use: Phase Category = "Billable — Research & Data" → Total UX team billable hours across all studies
Billable — Client Meetings & Support
Applies to: Monitor Products · Client Requests
Client Delivery & Support phases and quick-turnaround client request work. Time spent on client-facing activities, calls, and Q&A.
Filter use: Phase Category = "Billable — Client Meetings & Support" + Tag = "ad-hoc" → Total unplanned client burden
Non-Billable Categories
Non-Billable — Internal Operations
Applies to: All Internal Ops Projects
All gold/yellow project phases: Admin Work, Team Meetings, Training, and Marketing/BD Support. These are overhead costs, not client-billable.
Filter use: Phase Category = "Non-Billable — Internal Operations" → Total overhead hours for capacity planning
Non-Billable — Scoping & BD
Applies to: Monitor Products · Custom Projects (pre-engagement phases only)
Pre-sale and scoping work logged directly under the relevant monitor or custom project. For monitors, this means time in the Planning & Scoping phase before a deal is signed. Valuable for calculating BD cost per conversion without needing separate scoping projects.
Filter use: Phase Category = "Non-Billable — Scoping & BD" → Total BD investment by project; compare vs. deals closed
Non-Billable — Business Systems Management
Applies to: Business Systems Management project (Document Management team)
All work logged to the Business Systems Management project — Timeneye administration, Pardot administration, internal system reporting, and process documentation. Represents infrastructure investment, not client delivery work.
Filter use: Phase Category = "Non-Billable — Business Systems Management" → Total systems investment hours for budget planning and team capacity review
Time Off
Applies to: Time Off Project
Vacation, sick, holidays, personal days, parental leave, bereavement, jury duty, and work travel. No changes needed — this category is already working correctly.
Filter use: Phase Category = "Time Off" + User → Total time off per person for HR tracking

The Billing Audit Formula

  1. Filter Phase Category = any Non-Billable category AND Billing Status = "Billable" → These entries are misclassified. Review and correct.
  2. Filter Phase Category = any Billable category AND Billing Status = "Billed" → Confirmed billed hours. Use for revenue recognition.
  3. Filter Phase Category = any Billable category AND Billing Status = "Unbilled" → Hours earned but not yet invoiced. Prioritize for next billing cycle.

Tag Rules

Use lowercase, hyphenated format: ad-hoc not Ad-Hoc or adhoc
Apply 0–3 tags per entry. More than 3 usually means the entry should be split into separate time logs.
Tags are optional for internal/non-billable work unless specifically needed for reporting.
Do not combine tags from the same category on one entry — use the most specific one.
Never create new tags without admin approval. Unauthorized tags break cross-project filters.
If none of the 14 tags apply, leave the tag field empty — don't invent a new one.

Consistent naming is what makes bulk filtering, year-over-year comparisons, and billing audits actually work. These rules apply to every name in Timeneye — projects, phases, tags, and tasks. When in doubt: "Could someone else immediately understand what this is from the name alone?"
Core Naming Rules
✅ Title Case — Always
Capitalize the first letter of every major word. Applies to project names, phase names, tasks, and tags.

Bank Monitor 2026 ✓
bank monitor 2026 ✗
BANK MONITOR 2026 ✗

Exception: Short prepositions and conjunctions stay lowercase unless they begin the name.
✅ Pipe Delimiter ( | ) — Sparingly
Use space-pipe-space | to separate logical segments. Not in phase or task names.

Chase | Client Requests 2026 ✓
Chase|ClientRequests ✗

Limit to one or two pipes per name. Three or more signals the name is too complex.
✅ Year Formatting
Always use four-digit years at the end of project names. Never abbreviate.

Bank Monitor 2026 ✓
Bank Monitor '26 ✗
Bank Monitor 26 ✗

Internal ops projects (gold) are evergreen — no year needed.
⚠️ Phases Are Never Dated
Phase names must never contain dates, months, quarters, or deliverable-specific names. These belong in the Task field.

Content Production ✓ (phase)
January Report ✗ (phase)
January Report ✓ (task under Content Production)
⚠️ No Abbreviations in Project Names
Spell out the full monitor name in Timeneye. Abbreviations are for Salesforce only.

Bank Monitor 2026 ✓
BNK Monitor 2026 ✗
Credit Card Monitor 2026 ✓
CCM 2026 ✗ (Salesforce format only)
📏 Character Limits
Keep project names under 50 characters. Task names should be 3–6 words — descriptive but concise.

Allowed special characters: & in phases · - in tags only
Never use: # @ / \ : ; " in any name.

Before vs. After — Why These Filters Didn't Work Before

Every filter above was technically available in Timeneye before the restructuring. The reason they didn't work:

  • Phase Category was useless because phases weren't standardized — the same type of work had dozens of different phase names across projects, so filtering by category returned incomplete results.
  • Tag filters returned ~58% of entries because 42% had no tags at all. Reports showed only partial pictures.
  • Year-over-year filters broke because "Content Production" in 2025 might have been called "Report Writing," "January Production," "Writing & Research," or "Monitor Production" — there was no consistent name to filter against.
  • Billing audit filters were unreliable because billable/non-billable wasn't enforced at the phase level — it was set manually per entry and frequently wrong.

The restructuring doesn't add new filter options — it makes the existing filters actually work correctly.

Notes, Tags & Entry Details
Notes are not required but are encouraged whenever context might matter later. Good Notes make billing audits faster and protect you if a client ever questions a charge.

Write a note when:
  • The phase alone doesn't explain what you actually did ("Drafted Q3 benchmark exec summary")
  • The work was ad hoc or unplanned ("Emergency data pull for client request — received 4pm")
  • You're splitting time across multiple activities in one block
  • The work might be questioned in a billing review ("45 min call with JPMorgan team re: data discrepancy")
  • You're in a meeting with multiple stakeholders and want to record who was there
Skip the note when:
  • The project + phase combination already tells the complete story (Time Off → Vacation needs no note)
  • It's a routine recurring activity like a weekly team meeting
Keep notes factual and brief — one sentence is usually enough. Avoid vague entries like "worked on project" or "misc tasks."
Tags are optional and exist to enable cross-project reporting — they answer "how" the work was done or "who" contributed, beyond what the project and phase can tell you.

Use a tag when:
  • You want your work to surface in cross-project filters (e.g. all editorial work across all monitors)
  • The work was unplanned — always add ad-hoc
  • Channel matters for reporting — use web or mobile for research/production
  • You're contributing as a specialist — e.g. editorial, graphics
Don't: invent new tags or use free-text. Only use tags from the approved list — see the Tags & Tasks tab for the full list with descriptions.
Most routine entries don't need tags at all. Internal meetings, time off, and standard workflow phases can go without.
It depends on whether the work was in different projects or just different activities within the same project.

Different projects → split the entry. Time logged under the wrong project creates billing errors and can't be easily corrected later.

Same project, different phases → use your judgment. If you spent 2 hours writing and 1 hour in QA for the same monitor, splitting into two entries is ideal but not always practical. A single entry with a note like "Mix of content production and QA review" is acceptable for short blocks.
When in doubt, split. It takes 30 extra seconds and saves hours of audit time later.

It might feel simpler up front, but it would create billing problems. If editorial logs all their hours in a generic "Editorial Services" project, those hours disappear from the client project where they belong — making client billing reports incomplete and making it harder to see the true cost of each deliverable.

The Contributing Team tag approach gives you the best of both worlds: hours stay in the right client project for accurate billing, and filtering by tag gives you total cross-project visibility into each team's output — without any double-counting.
